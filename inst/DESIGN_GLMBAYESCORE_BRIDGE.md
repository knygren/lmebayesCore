# Design: `glmbayesCore` C-callable bridge (Stage 3c/3d)

Maintainer-facing implementation plan for the cross-package C++ interface that
lets `lmebayesCore`'s mixed-model C++ call `glmbayesCore`'s iid
samplers/envelope machinery directly, instead of compiling a private duplicate
copy of that engine. This is Stage 3c (build the bridge, in `glmbayesCore`)
and Stage 3d (consume it, in `lmebayesCore`) of the Stage 3 C++ dedup bridge
plan (maintainer's local Cursor plan store, not part of this repo).
**Parked here, not in `glmbayesCore`, because `glmbayesCore` is currently
under CRAN review and no implementation changes are to be made there until
it is accepted.** Once
implementation starts, the `glmbayesCore`-side content of this doc (Sections
2-6) should move into `glmbayesCore` itself (e.g. as its own
`inst/DESIGN_CCALLABLE_BRIDGE.md`) and this file can be trimmed to the
`lmebayesCore`-side consumption notes (Section 7) or retired. Canonical doc;
excluded from the source tarball via `.Rbuildignore`.

---

## 1. Purpose and scope

### Goal

Give `lmebayesCore` (and any other downstream package) a stable, ABI-safe way
to call six `glmbayesCore` C++ entry points **without duplicating their
source**, so the ~17 files that are only reachable through those six entry
points can be deleted from `lmebayesCore` (Stage 3e).

### The six bridge-surface entry points

Traced from every direct (non-R-callback) call that `lmebayesCore`'s
mixed-model-only `.cpp` files make into the iid engine (Stage 3a audit):

| # | Function | Declared in (`glmbayesCore`) | Defined in | Called from (`lmebayesCore`) |
|---|----------|-------------------------------|-------------|-------------------------------|
| 1 | `glmbayes::sim::rNormalGLM()` | `src/simfuncs.h` | `src/rNormalGLM.cpp` | `src/rNormalGLMBlocks.cpp` |
| 2 | `glmbayes::sim::rNormalReg()` | `src/simfuncs.h` | `src/rNormalReg.cpp` | `src/rNormalRegBlocks.cpp`, `src/twoBlockGibbs.cpp` (Block 2, 3 call sites) |
| 3 | `glmbayes::sim::rIndepNormalGammaReg()` | `src/simfuncs.h` | `src/rIndepNormalGammaReg.cpp` | `src/twoBlockGibbs.cpp` (Block 2 ING path, 3 call sites) |
| 4 | `glmbayes::env::EnvelopeBuild()` | `src/Envelopefuncs.h` | `src/EnvelopeBuild.cpp` | `src/block_rIndepNormalGammaReg.cpp` (`BlockEnvelopeBuild()`) |
| 5 | `glmbayes::env::EnvelopeDispersionBuild()` | `src/Envelopefuncs.h` | `src/EnvelopeDispersionBuild.cpp` | `src/block_rIndepNormalGammaReg.cpp` (`BlockEnvelopeDispersionBuild()`) |
| 6 | `glmbayes::rng::rnorm_ct()` | `src/rng_utils.h` | `src/rng_utils.cpp` | `src/block_rIndepNormalGammaReg.cpp` (2 call sites, inner per-coordinate loop) |

Everything these six functions call *internally* (`famfuncs_*`, `EnvelopeEval`
/ `EnvelopeSize` / `EnvelopeSort` / `Set_Grid` / `Set_LogP`, most of
`rng_utils`, `invgamma_ct`) becomes purely `glmbayesCore`-internal once
bridged, and is deleted from `lmebayesCore` in Stage 3e — that's the payoff:
narrow bridge surface (6 functions), large prunable footprint (~17 files).

### Out of scope

- Re-exporting these six functions as new **R-level** public API in
  `glmbayesCore` (see Section 3 — the mechanism chosen specifically avoids
  this).
- Bridging `EnvelopeCentering()`, `EnvelopeOrchestrator()`, or any other
  envelope routine not in the table above — current evidence (Stage 3a) says
  nothing in `lmebayesCore`'s mixed-model `.cpp` calls them directly. **Must
  be double-checked with a fresh direct-call grep at the start of 3c**, since
  `glmbayesCore` may have changed since the Stage 3a audit.
- Changing the numerical behavior of any of the six functions. This is a pure
  linking change: `lmebayesCore` calls the exact same code, compiled once, in
  `glmbayesCore`'s DLL instead of its own.

## 2. Why not just re-export at the R level?

`lmebayesCore` already gets R-level access to `glmbayesCore`'s iid stack via
`Imports: glmbayesCore` (Stage 0/1). That is *not* sufficient here because the
six entry points above are called **at the C++ level, from inside
`lmebayesCore`'s own compiled `.cpp` files**, mid-algorithm (e.g. once per
Block-1 group per Gibbs sweep) — not once per top-level R call. Round-tripping
through R (`Rcpp::Function`, `.Call`) for every inner-loop invocation would
mean SEXP marshaling and R-evaluator overhead on a hot path that currently
runs in pure C++. A C-callable bridge keeps the call in compiled code end to
end; only the *first* call per process pays a one-time `R_GetCCallable()`
lookup (cached in a `static` pointer, same pattern already used in
[`src/R_interface.h`](../src/R_interface.h) for R-callback lookups).

---

## 3. Mechanism: `Rcpp::interfaces(cpp)`, not hand-rolled `R_RegisterCCallable`

`glmbayesCore`'s `src/RcppExports.cpp` (and its `R_init_glmbayesCore()`) is
**generated by `Rcpp::compileAttributes()`** — do not hand-edit it; anything
added by hand is silently overwritten on the next `compileAttributes()` run.
Rcpp Attributes has a built-in mechanism for exactly this cross-package C++
sharing use case, and it is strongly preferred over hand-writing
`R_RegisterCCallable()` / `R_GetCCallable()` boilerplate:

```cpp
// [[Rcpp::export]]
// [[Rcpp::interfaces(cpp)]]
Rcpp::List rNormalGLM(int n, NumericVector y, /* ... */, bool verbose = false) {
  // unchanged body
}
```

- `// [[Rcpp::interfaces(cpp)]]` tells `compileAttributes()` to generate the
  **C++-callable interface only** — it does **not** create a new R-level
  wrapper function or NAMESPACE export. (`interfaces(r, cpp)` would do both;
  we specifically want `cpp`-only so this stays an internal implementation
  detail, per the repo's public-API-changes rule — no new exports.)
- `compileAttributes()` then:
  1. Registers the function via `R_RegisterCCallable("glmbayesCore",
     "rNormalGLM", (DL_FUNC) &rNormalGLM_wrapper)` — folded automatically into
     the existing auto-generated `R_init_glmbayesCore()` in `RcppExports.cpp`.
  2. Generates a header (expected path: `inst/include/glmbayesCore_RcppExports.h`
     — **confirm exact filename on the first `compileAttributes()` run**, it
     has varied slightly across Rcpp versions) containing an inline
     `glmbayesCore::rNormalGLM(...)` wrapper that does a lazily-cached
     `R_GetCCallable()` lookup, `Rcpp::wrap()`s the arguments, calls through,
     and `Rcpp::as<>()`s the result back — same signature as the original
     function, so **call sites need zero argument-shape changes**.
- Downstream (`lmebayesCore`) gets this via `LinkingTo: glmbayesCore` (Section
  7), which adds `glmbayesCore`'s `inst/include/` to the include path.

This only works cleanly for functions whose parameters/return type are
`Rcpp::wrap()`/`Rcpp::as()`-compatible — true for entries 1-5 (all
`Rcpp::List`, `NumericVector`, `NumericMatrix`, `std::string`, `Rcpp::Function`,
scalar `bool`/`int`/`double`). It is **not** the right tool for entry 6
(`rnorm_ct`) — see Section 5.

### Open risk: functions live inside `namespace glmbayes { namespace sim {…} }` / `namespace glmbayes { namespace env {…} }`

All five `Rcpp::List` entry points are defined inside nested C++ namespaces,
not at file (global) scope. Modern Rcpp Attributes is documented to handle
namespaced `// [[Rcpp::export]]` functions, but this repo hasn't verified it
against `glmbayesCore`'s specific `namespace glmbayes { namespace sim { … } }`
nesting. **First task in 3c:** attempt `compileAttributes()` with the
attribute placed directly on the existing namespaced definitions; if it
mishandles the namespace (wrong qualified name in the generated registration,
or a compile error), fall back to a thin free-function forwarder placed at
global scope in each file, tagged with the export attributes, e.g.:

```cpp
// [[Rcpp::export]]
// [[Rcpp::interfaces(cpp)]]
inline Rcpp::List rNormalGLM_bridge(int n, NumericVector y, /* ... */) {
  return glmbayes::sim::rNormalGLM(n, y, /* ... */);
}
```

(with the generated C++ interface then named `rNormalGLM_bridge` in the
`glmbayesCore::` namespace instead of `rNormalGLM` — a one-line rename at
`lmebayesCore` call sites in 3d either way).

## 5. `rnorm_ct()`: hand-rolled bridge, not `Rcpp::interfaces`

```cpp
double glmbayes::rng::rnorm_ct(double lgrt, double lglt, double mu, double sigma);
```

Pure POD signature (`double(double,double,double,double)`), called in a tight
per-coordinate loop inside `BlockEnvelopeBuild`'s / Block~1's Gibbs sweep in
`lmebayesCore`'s `src/block_rIndepNormalGammaReg.cpp` — potentially many
thousands of calls per posterior draw. Routing this through
`Rcpp::interfaces(cpp)` would work correctly but forces every call through
`Rcpp::wrap()` (REALSXP allocation) on the way in and `Rcpp::as<double>()` on
the way out — R-heap allocation and GC pressure on a hot path that currently
never touches the R heap. Bypass that entirely with a raw function pointer:

**In `glmbayesCore` (`src/rng_utils.cpp`), add:**

```cpp
// [[Rcpp::init]]
void glmbayesCore_register_rnorm_ct(DllInfo *dll) {
  R_RegisterCCallable(
    "glmbayesCore", "rnorm_ct",
    (DL_FUNC) &glmbayes::rng::rnorm_ct
  );
}
```

`// [[Rcpp::init]]` is the Rcpp-Attributes-supported hook for exactly this: it
gets folded into the auto-generated `R_init_glmbayesCore()` alongside the
`interfaces(cpp)` registrations from Section 4, without ever hand-editing
`RcppExports.cpp`.

**New header `glmbayesCore/inst/include/glmbayesCore_rnorm_ct.h`** (hand-written,
not generated — mirrors the caching pattern in `lmebayesCore`'s
[`src/R_interface.h`](../src/R_interface.h)):

```cpp
#ifndef GLMBAYESCORE_RNORM_CT_H
#define GLMBAYESCORE_RNORM_CT_H

#include <R_ext/Rdynload.h>

namespace glmbayesCore {

inline double rnorm_ct(double lgrt, double lglt, double mu, double sigma) {
  typedef double (*Ptr_rnorm_ct)(double, double, double, double);
  static Ptr_rnorm_ct fn = NULL;
  if (fn == NULL) {
    fn = (Ptr_rnorm_ct) R_GetCCallable("glmbayesCore", "rnorm_ct");
  }
  return fn(lgrt, lglt, mu, sigma);
}

}  // namespace glmbayesCore

#endif
```

This header needs to be `#include`d explicitly by consumers (it won't be
pulled in automatically by the `Rcpp::interfaces`-generated header from
Section 3) — document this in the `glmbayesCore` `inst/include/README` (or
top of the header itself) so it isn't lost track of.

## 6. Verifying the bridge in isolation (within `glmbayesCore`, before touching `lmebayesCore`)

Once 3c lands, verify from *inside* `glmbayesCore` before any `lmebayesCore`
change:

1. `Rcpp::compileAttributes()` then `devtools::load_all()` / `R CMD INSTALL`
   — confirm no new NAMESPACE exports appear (`git diff NAMESPACE` should be
   empty).
2. A throwaway `data-raw/` script in `glmbayesCore` that writes a tiny
   downstream test package (or reuses `lmebayesCore`'s dev tree pre-3d) with
   `LinkingTo: glmbayesCore`, `#include <glmbayesCore_RcppExports.h>` +
   `#include <glmbayesCore_rnorm_ct.h>`, and one call to each of the six
   bridged functions — confirm it compiles and returns results identical to
   calling the local (pre-bridge) copies with the same seed.
3. `devtools::check()` on `glmbayesCore` itself — the new `interfaces(cpp)` /
   `Rcpp::init` attributes must not introduce any `R CMD check` NOTE/WARNING.

## 7. Consuming the bridge from `lmebayesCore` (Stage 3d — blocked on CRAN via 3c)

### `DESCRIPTION`

Add `glmbayesCore` to `LinkingTo` (currently `Rcpp, RcppArmadillo,
RcppParallel`):

```diff
-LinkingTo: Rcpp, RcppArmadillo, RcppParallel
+LinkingTo: Rcpp, RcppArmadillo, RcppParallel, glmbayesCore
```

`Imports: glmbayesCore (>= 0.5.1)` is already present (Stage 0) — bump the
version floor to whatever `glmbayesCore` release first ships the bridge.

### Call-site retargeting (mechanical, no logic change)

Because the bridged signatures are byte-identical to the local copies being
deleted (verified in the table in Section 1 — `lmebayesCore`'s
`rNormalReg()`, `EnvelopeBuild()`, etc. are, as of today, textually identical
declarations to `glmbayesCore`'s, since both trace back to the same fork
point), retargeting is: swap the `#include` and add the namespace qualifier.

| File | Before | After |
|------|--------|-------|
| `src/twoBlockGibbs.cpp` | `#include "simfuncs.h"`; unqualified `rNormalReg(...)`, `rIndepNormalGammaReg(...)` (resolves via `namespace glmbayes { namespace sim {` enclosing the file) | `#include <glmbayesCore_RcppExports.h>`; `glmbayesCore::rNormalReg(...)`, `glmbayesCore::rIndepNormalGammaReg(...)` |
| `src/rNormalGLMBlocks.cpp` | unqualified `rNormalGLM(...)` | `glmbayesCore::rNormalGLM(...)` |
| `src/rNormalRegBlocks.cpp` | unqualified `rNormalReg(...)` | `glmbayesCore::rNormalReg(...)` |
| `src/block_rIndepNormalGammaReg.cpp` | unqualified `EnvelopeBuild(...)`, `EnvelopeDispersionBuild(...)`, `rnorm_ct(...)` | `#include <glmbayesCore_RcppExports.h>` + `#include <glmbayesCore_rnorm_ct.h>`; `glmbayesCore::EnvelopeBuild(...)`, `glmbayesCore::EnvelopeDispersionBuild(...)`, `glmbayesCore::rnorm_ct(...)` |

If Section 3's namespace risk materializes and a `_bridge`-suffixed forwarder
is needed in `glmbayesCore`, the qualified names above become
`glmbayesCore::rNormalGLM_bridge(...)` etc. — same mechanical swap, just a
different name.

**No change** to argument construction, return-value handling, RNG seeding,
or control flow at any of these call sites — this is purely "which compiled
copy answers the call."

### `Makevars` / `Makevars.win`

Should need **no manual change**: `LinkingTo` handles the include path
automatically via `Rcpp:::LdFlags()` / `Rcpp:::CxxFlags()` machinery that
`R CMD SHLIB` invokes. Confirm this empirically in 3d — if `glmbayesCore`
ships any custom OpenCL/CUDA linker flags in its own `Makevars` that need to
propagate transitively (unlikely, since we're only linking against
`inst/include` headers, not `glmbayesCore`'s compiled `.o`/`.so` — this is a
*header-only* C-callable bridge, no new link-time library dependency), note
and resolve here.

## 9. Testing plan (3d/3f)

- **Bit-for-bit equivalence of deterministic outputs:** `EnvelopeBuild()`'s
  grid geometry, `EnvelopeDispersionBuild()`'s `disp_lower`/`disp_upper`, and
  any posterior-mode (`coef.mode`) computation are deterministic — must match
  the pre-bridge local-copy output exactly, same inputs.
- **Draw-level equivalence only in distribution, not per-draw:** rejection
  samplers (`rNormalReg`, `rIndepNormalGammaReg`, `rnorm_ct`) consume their
  own RNG stream position: **do not** assert per-draw equality against the
  pre-bridge copies; compare draw means/sds over long runs, as the existing
  `data-raw/test_two_block_cpp.R`-style scripts already do for other
  port-only stages.
- **Full regression suite:** all of `tests/testthat/`, especially
  `test-rIndepNormalGammaRegBlock-iters-parity*.R` (Block~1 ING envelope
  path — the heaviest user of entries 4-6).
- **`devtools::check()`** on both `glmbayesCore` (bridge additions) and
  `lmebayesCore` (consumer changes + Stage 3e deletions) — 0 errors/warnings
  before merging.
- **OpenCL smoke test** if a GPU is available: `EnvelopeBuild()`'s OpenCL
  dispatch path is part of what's being bridged; confirm it still dispatches
  correctly when invoked through `R_GetCCallable()` rather than a direct
  intra-DLL call.

## 10. Open questions

- Exact generated header filename/path for `Rcpp::interfaces(cpp)` output —
  confirm on first `compileAttributes()` run against the installed Rcpp
  version (Section 3).
- Whether the namespace nesting (`glmbayes::sim::`, `glmbayes::env::`)
  requires the forwarder fallback (Section 3) — resolve at the start of 3c.
- Whether to also expose this bridge as a *documented*, stable public C++ API
  for other downstream packages beyond `lmebayesCore` (out of scope for this
  pass — treat as an internal, unstable-ABI consumption path for now, same
  caveat CRAN expects for `LinkingTo`-only headers).
- Version floor: pin `lmebayesCore`'s `Imports: glmbayesCore (>= X.Y.Z)` /
  `LinkingTo: glmbayesCore` to whatever version first ships the bridge, once
  known.
